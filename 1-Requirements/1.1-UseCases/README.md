## Описи варіантів використання програмного продукту

### **UC001: Керування користувачами системи**

| Параметр | Значення |
|----------|----------|
| **Ідентифікатор та назва** | UC001: Керування користувачами системи |
| **Автор** | Костецький А.В. |
| **Дата створення** | 2025 |
| **Основна дійова особа** | Системний адміністратор |
| **Додаткова дійова особа** | **Керівник IT-відділу, Користувач системи** |
| **Опис** | Додавання нових користувачів, редагування їх прав доступу та видалення облікових записів в системі моніторингу закупівель |
| **Тригер** | Надходження заявки від **користувача** на реєстрацію; Рішення **керівника IT-відділу** про зміну прав доступу |
| **Попередні умови** | Системний адміністратор авторизований в системі; Має права адміністратора |
| **Нормальний напрямок подій** | 1. Системний адміністратор отримує заявку від **користувача** або вказівку від **керівника IT-відділу**<br>2. Переходить до розділу "Керування користувачами"<br>3. Обирає операцію (додати/редагувати/видалити/деактивувати)<br>4. Заповнює дані користувача (ПІБ, пошта, відділ, роль, рівень доступу)<br>5. Система перевіряє унікальність email та коректність даних<br>6. Призначає права доступу відповідно до ролі (Менеджер/Аналітик)<br>7. Зберігає зміни та надсилає запрошення **користувачу**<br>8. Система логує операцію в журналі аудиту |
| **Альтернативний напрямок подій** | 5а. Якщо email вже існує - система пропонує використати інший<br>7а. Якщо користувач деактивований - система блокує доступ, але зберігає історію дій |
| **Винятки** | Конфлікт даних; Втрата з'єднання; Спроба видалення активної заявки |
| **Частота використання** | 1-2 рази на місяць |
| **Бізнес правила** | BR001: Мінімум один активний системний адміністратор<br>BR002: Заборона призначення ролі "Системний адміністратор" без додаткового затвердження **керівником IT-відділу** |
| **Спеціальні вимоги** | Двофакторна автентифікація для системних адміністраторів; Архівування історії змін протягом 5 років |
| **Припущення** | Системний адміністратор пройшов відповідне навчання та має необхідні повноваження |

### **UC002: Перегляд аналітики закупівель**

| Параметр | Значення |
|----------|----------|
| **Ідентифікатор та назва** | UC002: Перегляд аналітики закупівель |
| **Автор** | Костецький А.В. |
| **Дата створення** | 2025 |
| **Основна дійова особа** | Аналітик |
| **Додаткова дійова особа** | **Керівник відділу закупівель, Фінансовий директор** |
| **Опис** | Огляд статистичних даних, звітів та аналітики закупівель без можливості внесення змін для аналізу ефективності |
| **Тригер** | Необхідність аналізу ефективності закупівель; Запит від **керівника відділу закупівель**; Підготовка звітів для **фінансового директора** |
| **Попередні умови** | Аналітик авторизований в системі; Має роль "Аналітик" |
| **Нормальний напрямок подій** | 1. Аналітик отримує запит від **керівника відділу закупівель** або **фінансового директора**<br>2. Переходить до розділу "Аналітика та звіти"<br>3. Обирає тип аналітики (фінансові показники, ефективність постачальників, тенденції цін)<br>4. Встановлює період аналізу та параметри фільтрації<br>5. Система відображає дані у вигляді інтерактивних графіків, теплових карт та таблиць<br>6. Аналітик використовує фільтри для деталізації аналізу<br>7. Експортує результати у форматі PDF/Excel та надає **керівництву** |
| **Альтернативний напрямок подій** | 5а. Якщо дані відсутні - система показує повідомлення з рекомендаціями щодо періоду аналізу<br>7а. Якщо звіт великий - система пропонує асинхронне формування з подальшим сповіщенням **керівника відділу** |
| **Винятки** | Відсутність даних для обраного періоду; Обмежений доступ до конфіденційних даних |
| **Частота використання** | Щоденно (моніторинг) + щомісяця (звітність) |
| **Бізнес правила** | BR003: Автоматичне оновлення аналітичних даних кожні 4 години<br>BR004: Збереження історії аналітики протягом 3 років |
| **Спеціальні вимоги** | Read-only доступ; Можливість створення персоналізованих dashboard; Експорт у стандартних форматах |
| **Припущення** | Аналітик має необхідні знання для інтерпретації аналітичних даних |

### **UC003: Пошук та порівняння постачальників**

| Параметр | Значення |
|----------|----------|
| **Ідентифікатор та назва** | UC003: Пошук та порівняння постачальників |
| **Автор** | Костецький А.В. |
| **Дата створення** | 2025 |
| **Основна дійова особа** | Менеджер з закупівель |
| **Додаткова дійова особа** | **Головний інженер, Керівник проекту** |
| **Опис** | Пошук постачальників будівельного обладнання за критеріями, порівняння пропозицій та збереження результатів для подальшого використання |
| **Тригер** | Необхідність закупівлі нового обладнання; Технічні вимоги від **головного інженера**; Запит від **керівника проекту** |
| **Попередні умови** | Менеджер з закупівель авторизований в системі; Має роль "Менеджер з закупівель" |
| **Нормальний напрямок подій** | 1. Менеджер отримує технічні вимоги від **головного інженера**<br>2. Вводить критерії пошуку (тип обладнання, технічні параметри, бюджет, регіон)<br>3. Система відображає список відповідних постачальників з рейтингами та історією поставок<br>4. Менеджер порівнює ціни, умови оплати, терміни поставки, гарантії<br>5. Консультується з **головним інженером** щодо технічних аспектів<br>6. Додає обраних постачальників до списку порівняння<br>7. Аналізує відгуки та історію співпраці з постачальниками<br>8. Надає результати **керівнику проекту** для подальшого затвердження |
| **Альтернативний напрямок подій** | 4а. Якщо результатів багато - менеджер використовує розширені фільтри (рейтинг, час роботи на ринку, наявність сертифікатів)<br>4б. Якщо постачальників немає - система пропонує розширити критерії пошуку або знайти аналогічне обладнання |
| **Винятки** | Відсутність постачальників за заданими критеріями; Технічна помилка пошукового модуля |
| **Частота використання** | Щоденно (активні закупівлі) |
| **Бізнес правила** | BR005: Пріоритет постачальникам з рейтингом вище 4.2 та стажем роботи понад 3 роки<br>BR006: Обов'язкове порівняння мінімум 3 постачальників для закупівель понад 50,000 грн |
| **Спеціальні вимоги** | Можливість збереження шаблонів пошуку; Інтеграція з ринковими даними в реальному часі |
| **Припущення** | База постачальників регулярно оновлюється та містить актуальну інформацію |

### **UC004: Формування заявки на закупівлю**

| Параметр | Значення |
|----------|----------|
| **Ідентифікатор та назва** | UC004: Формування заявки на закупівлю |
| **Автор** | Костецький А.В. |
| **Дата створення** | 2025 |
| **Основна дійова особа** | Менеджер з закупівель |
| **Додаткова дійова особа** | **Керівник проекту, Фінансовий контролер, Системний адміністратор** |
| **Опис** | Створення, заповнення та відправлення заявки на закупівлю будівельного обладнання для затвердження |
| **Тригер** | Вибір оптимального постачальника після порівняння; Необхідність оформити закупівлю для будівельного проекту; Затвердження від **керівника проекту** |
| **Попередні умови** | Менеджер з закупівель авторизований в системі; Завершено пошук та порівняння постачальників; Визначено бюджет |
| **Нормальний напрямок подій** | 1. Менеджер обирає постачальника зі збережених результатів пошуку<br>2. Заповнює форму заявки (обладнання, специфікації, кількість, вартість, терміни поставки)<br>3. Консультується з **керівником проекту** щодо пріоритетів<br>4. Додає обґрунтування вибору постачальника (ціна, якість, терміни)<br>5. Завантажує необхідні документи (технічне завдання, кошторис, специфікації)<br>6. Вказує бюджетний код та проект для віднесення витрат<br>7. Надсилає заявку на затвердження **керівнику проекту**, **фінансовому контролеру** та **системному адміністратору**<br>8. Система надсилає підтвердження, присвоює унікальний номер та відстежує статус заявки |
| **Альтернативний напрямок подій** | 7а. Якщо сума закупівлі менша за ліміт - система автоматично затверджує заявку<br>7б. Якщо документи відсутні - система дозволяє відправити заявку з позначкою "Документи додадуться" та встановлює дедлайн |
| **Винятки** | Постачальник не пройшов верифікацію; Перевищення виділеного бюджету; Відсутність необхідних документів; Конфлікт з графіком поставок |
| **Частота використання** | 3-5 разів на тиждень (залежно від обсягу проектів) |
| **Бізнес правила** | BR007: Автоматичне затвердження заявок до 25,000 грн<br>BR008: Максимальний термін розгляду заявки - 2 робочих дні<br>BR009: Обов'язкове затвердження фінансовим директором для закупівель понад 100,000 грн |
| **Спеціальні вимоги** | Можливість збереження чернетки заявки; Система сповіщень про зміну статусу; Інтеграція з системою документообігу |
| **Припущення** | Менеджер має всю необхідну інформацію та повноваження для формування заявки |
